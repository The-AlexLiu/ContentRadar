# ContentRadar

**ContentRadar 是一个面向内容研究场景的公开内容采集与分析工具。**

我把它定位成一个内容运营和竞品研究场景下的“数据雷达”：围绕关键词、帖子、创作者和评论，把散落在不同内容平台上的公开信息结构化采集下来，方便后续做选题分析、用户反馈整理、评论洞察、竞品内容跟踪和素材方向判断。

> 使用边界：本项目仅用于学习、研究和个人分析场景，不用于商业化采集、大规模爬取、绕过平台限制、平台干扰或任何违反平台规则的行为。使用前请阅读仓库中的 `LICENSE` 和下方免责声明。

## 项目简介

ContentRadar 是一个多平台公开内容采集工具，覆盖小红书、抖音、快手、B 站、微博、贴吧、知乎等内容平台。它不是单个平台的脚本集合，而是把不同平台的搜索、详情、评论、创作者主页、登录态和数据保存方式统一到一套命令入口里。

我更关注它在内容运营和增长分析里的实际价值：当需要判断一个选题有没有热度、一个竞品账号最近在推什么内容、用户在评论区反复提到什么问题、广告素材可以从哪些真实表达里找角度时，ContentRadar 可以先把公开内容结构化沉淀下来，再交给表格、数据库或后续分析流程处理。

它目前覆盖的典型采集场景包括：

| 采集场景 | 可以拿到什么 |
|---|---|
| 关键词搜索 | 按关键词采集帖子、视频、回答、贴吧帖子等内容列表 |
| 指定帖子 ID | 采集指定内容的详情、互动数据和评论 |
| 一级 / 二级评论 | 获取用户反馈、问题、吐槽、购买动机和高频表达 |
| 创作者主页 | 跟踪指定账号的公开内容和更新节奏 |
| 多平台横向对比 | 对比同一关键词在不同平台的内容形态和互动表现 |
| 数据导出 | 保存为 CSV、JSON、JSONL、Excel、SQLite、MySQL 等格式 |

## 核心设计

### 1. 基于浏览器自动化保存登录态

ContentRadar 基于 Playwright 和浏览器上下文运行，可以复用用户主动登录后的页面环境，降低处理复杂前端参数的成本。

### 2. 利用浏览器上下文处理平台签名

很多内容平台的请求都依赖前端页面生成的签名参数。如果完全从接口层重新实现，通常需要 JS 逆向和加密逻辑还原，维护成本很高。

ContentRadar 的思路更偏工程化：先通过 Playwright 打开真实浏览器页面，再在浏览器上下文中执行必要的 JS 表达式，获得页面正常运行需要的参数。这样可以尽量复用真实页面环境，减少手工逆向复杂加密算法的工作量。

这不代表可以绕过平台限制。登录态、代理配置和浏览器自动化都应该只用于用户有权访问、平台规则允许、频率可控的研究场景。

### 3. 通过统一命令切换平台

不同平台共用统一入口，通过 `--platform`、`--type`、`--lt` 等参数控制采集平台、登录方式和采集模式。

```bash
uv run main.py --platform xhs --lt qrcode --type search
uv run main.py --platform xhs --lt qrcode --type detail
uv run main.py --help
```

### 4. 提供 WebUI 可视化操作

除了命令行方式，也可以启动 WebUI，在页面中配置平台、登录方式、采集类型，并查看运行状态。

```bash
uv run uvicorn api.main:app --port 8080 --reload
```

启动后访问：

```text
http://localhost:8080
```

## 技术原理

ContentRadar 的核心链路可以拆成四层：

| 层级 | 作用 |
|---|---|
| 命令入口 | `main.py` 读取 `--platform`、`--type`、`--lt`、`--save_data_option` 等参数，决定本次任务的目标平台、采集类型和保存方式 |
| 平台适配层 | `media_platform/` 下按平台拆分实现，例如小红书、抖音、B 站、微博、贴吧、知乎等，每个平台负责自己的登录、搜索、详情、评论和创作者逻辑 |
| 浏览器与签名层 | 通过 Playwright、CDP、本地 Chrome、Cookie 和页面上下文处理登录态与签名参数，减少直接逆向平台加密逻辑的成本 |
| 数据存储层 | `store/` 和 `database/` 负责把采集结果写入 CSV、JSON、JSONL、Excel、SQLite、MySQL 等目标格式 |

如果放到内容运营场景里，我会把它理解成一个“公开内容数据采集底座”：前面负责拿数据，中间负责统一字段和平台差异，后面负责把数据沉淀到可以分析的格式里。后续无论是做选题表、评论聚类、竞品内容库，还是广告素材洞察，都可以基于这些结构化数据继续处理。

## 快速开始

### 1. 安装 uv

推荐使用 `uv` 管理 Python 环境：

```bash
uv --version
```

如果尚未安装，可以参考 [uv 官方安装指南](https://docs.astral.sh/uv/getting-started/installation)。

### 2. 安装 Node.js

部分平台能力依赖 Node.js，建议安装 `>= 16.0.0`。

### 3. 安装依赖

```bash
uv sync
```

### 4. 配置 Chrome CDP 模式

默认推荐使用 CDP 模式连接本地 Chrome 浏览器：

1. 安装最新版 Chrome；
2. 打开 `chrome://inspect/#remote-debugging`；
3. 勾选 `Allow remote debugging for this browser instance`；
4. 确认页面显示 `Server running at: 127.0.0.1:9222`。

如果不使用 CDP 模式，可以在 `config/base_config.py` 中设置：

```python
ENABLE_CDP_MODE = False
```

### 5. 运行采集任务

```bash
# 关键词搜索
uv run main.py --platform xhs --lt qrcode --type search

# 指定帖子详情和评论
uv run main.py --platform xhs --lt qrcode --type detail

# 查看完整参数
uv run main.py --help
```
